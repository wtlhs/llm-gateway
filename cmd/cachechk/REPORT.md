# Prompt Caching 影响验证

> 生成日期: 2026-06-30
> 验证对象: 网关是否会破坏上游 LLM 的 prompt 前缀缓存命中
> 结论: **✅ 不破坏**(经网关 4 次, 缓存命中率始终 98%, 与直连一致)

## 为什么要验证

New API v0.13.2 + MiniMax-M3 响应里含 `"prompt_tokens_details":{"cached_tokens":...}`,
说明上游启用了 **prompt 前缀缓存**(相同 system prompt 前缀复用, 降费提速)。

缓存命中通常基于**请求体的哈希/前缀**。如果网关在透传时改变了请求体字节序
(如重新序列化 JSON), 会破坏缓存键, 导致缓存失效 → 费用上升 + 延迟增加。

## 方法

用相同的较长 system prompt(重复 40 次的固定文本, ~613 tokens)连发请求,
观察 `cached_tokens / prompt_tokens` 命中率:

1. 直连 New API 预热 2 次(建立上游缓存)
2. **经网关** 4 次(关键: 缓存是否被破坏)
3. 回直连 2 次(对照)

运行:
```bash
NEWAPI_LIVE_URL=https://newapi.wtlhs.com \
NEWAPI_LIVE_TOKEN=sk-xxx \
NEWAPI_LIVE_MODEL=MiniMax-M3 \
go run ./cmd/cachechk/
```

## 实测结果

```
=== 预热: 直连 2 次 ===
  直连第1次: cached=599/613 (98%)
  直连第2次: cached=599/613 (98%)

=== 经网关 4 次(关键: 缓存是否被网关破坏) ===
  网关第1次: cached=599/613 (98%)
  网关第2次: cached=599/613 (98%)
  网关第3次: cached=599/613 (98%)
  网关第4次: cached=599/613 (98%)

=== 对照: 回直连 2 次 ===
  直连第1次: cached=599/613 (98%)
  直连第2次: cached=599/613 (98%)
```

**经网关 4 次, 命中率始终 98%, 与直连完全一致。缓存未被破坏。**

## 为什么不破坏

网关采用**原字节透传**(DESIGN.md K3 设计):
- `snapshotBody` 读取请求体 → 解压副本用于落库
- `restoreBody` 用**原始压缩字节**(不是重新序列化的)还原 `r.Body`
- 上游收到的请求体字节序列与直连**完全一致**
- 因此上游的缓存键(基于请求体哈希/前缀)不变

## 结论

| 维度 | 影响 |
|---|---|
| prompt 缓存命中 | ✅ 不影响(98% 命中, 与直连一致) |
| 请求体字节序 | ✅ 原样透传(不重新序列化) |
| token 计费 | ✅ 不影响(usage 原样透传) |
| 注入的 X-Ctx-Gateway-Id 头 | ✅ 不影响缓存(上游不基于此头做缓存键) |

## 注意

- 缓存命中基于请求体内容, 网关**绝不能**对请求体做任何改写(包括字段顺序、空白)。
  当前实现满足, 但未来若加 Phase 3 上下文注入(改写 prompt), **会破坏缓存** ——
  需在注入逻辑里评估缓存影响, 或仅注入未缓存部分。
