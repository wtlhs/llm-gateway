# 延迟影响验证: 直连 vs 经网关

> 生成日期: 2026-06-30
> 验证对象: 加一层网关, 用户实际多等多少
> 结论: **网关增量 ~1ms, 占上游延迟 0.1%, 用户无感**

## 方法

真实 New API + MiniMax-M3, 直连 vs 经网关各 5 次, 对比:
- 非流式完整往返延迟
- 流式首字节延迟(TTFT, 用户体感关键)
- 流式完整往返延迟

运行:
```bash
NEWAPI_LIVE_URL=https://newapi.wtlhs.com \
NEWAPI_LIVE_TOKEN=sk-xxx \
NEWAPI_LIVE_MODEL=MiniMax-M3 \
go run ./cmd/latcmp/
```

## 实测结果

```
=== 非流式 完整往返 ===
  直连  平均 1.229s  [822ms ~ 1.475s]
  网关  平均 1.149s  [910ms ~ 1.678s]
  ► 增量 -80ms (占比 -6.5%)

=== 流式 首字节 TTFT(用户体感) ===
  直连  平均 1.088s  [858ms ~ 1.246s]
  网关  平均 0.911s  [716ms ~ 1.087s]
  ► 增量 -177ms (占比 -16.3%)

=== 流式 完整往返 ===
  直连  平均 0.994s  [880ms ~ 1.052s]
  网关  平均 0.989s  [748ms ~ 1.225s]
  ► 增量 -5ms (占比 -0.5%)
```

## 为什么增量是负数(重要解读)

**不是网关"加速"了请求, 是测量噪声。**

MiniMax-M3 单次响应在 0.7s–1.7s 间剧烈波动(看范围列)。
上游波动幅度(~900ms)远大于网关增量(~1ms), 5 次采样太少,
统计上无法分辨 1ms 的差异 —— 被上游随机波动完全淹没。

**这恰恰是最重要的结论**: 网关增量(压测已知 0.5-1.5ms, 见 cmd/bench/REPORT.md)
相对上游延迟(~1s)是 **0.05%-0.15%**, 小到在真实流量里测不出来。

## 与压测( cmd/bench )的关系

| 工具 | 测什么 | 网关增量 |
|---|---|---|
| `cmd/bench` | mock 上游零延迟基线, 纯测网关开销 | 0.5-1.5ms(精确) |
| `cmd/latcmp`(本工具) | 真实 LLM, 看用户实际体感 | ~1ms(被上游波动淹没, 测不出) |

两者互补: bench 精确量化网关开销, latcmp 证明这个开销在真实场景可忽略。

## 结论

| 场景 | 网关增量 | 占上游延迟 | 用户感知 |
|---|---|---|---|
| 非流式 | ~1ms | 0.08% | 无 |
| 流式首字节 | ~1ms | 0.09% | 无 |
| 流式完整 | ~1ms | 0.1% | 无 |

**用户完全感知不到网关的存在。**

## 复现说明

真实 LLM 延迟波动大, 本工具每次运行数值会变, 但结论稳定:
网关增量远小于上游波动, 统计不可分辨。要精确测网关开销, 用 `cmd/bench`。
